Assignment

Overview

Your assignment submission should be a HTML document that you have compiled using R Markdown. You don’t need to separately include the Rmd file (i.e. the code file) but you do need to use code folding so that we can inspect your code where required. Using code folding is also super handy to use as a check if your report is well written - your report should make sense and provide all the relevant information in the text when the code is hidden. Don’t rely on the reader to understand your code and any output needs to be explained in the text.

It should be written in a report style as if you were analysing a data set for a client. Below we provide some questions as guidance for what your report should contain.

  • Think about how your report might be structured (e.g. introduction, results, conclusion).
  • Do your figures have captions?
  • Are your figures legible? Are they too big?
  • Are you using headings and sub-headings effectively in your report?
  • Have you included sufficient and appropriate citations? This includes software, data, and other reference material.
  • Is it well presented (e.g. no unnecessary warnings or messages showing up) is there sufficient text explaining what is being presented or are you relying too much on the reader being able to understand and interpret the code and R output.

You can work on this in your computer lab in consultation with others but you will need to submit your own report. Your tutor can also provide feedback on your approach.

Required questions

The data set we will consider comes from the class survey. The raw data can be downloaded here. The following questions need to be addressed in the report.

  1. Is this a random sample of DATA2X02 students?
  2. What are the potential biases? Which variables are most likely to be subjected to this bias?
  3. Are there any questions that needed improvement to generate useful data?
  4. Does the number of COVID tests follow a Poisson distribution?
  5. Perform two other hypothesis tests. Give some rationale for why you selected these hypothesis tests and interpret the results. Be sure to mention any limitations in the data that may impact your findings. You’re welcome to use any tests we have covered in class up to and including material in weeks 4 and 5 (try to have some variety in your selected tests).

Additional requirements for DATA2902 students will be posted to their resources page.

Is there evidence?

Where possible the framework for hypothesis testing outlined in the lectures should be used. You should be explicit about which test you are performing. Most of the quantitative tests that are presented require some assumptions and these should be checked before a valid conclusion can be drawn.

When performing a test, you should think about the assumptions and justify your chosen test statistic. Make sure to:

  1. State the null and alternative hypotheses.
  2. Use visulisations to comment on the data and/or assumptions.
  3. Perform a suitable test (including checking any relevant assumptions and, where appropriate, justify why you chose that test over others that test the same hypothesis).
  4. Report the test statistic and p-value.
  5. Draw your conclusion about the null hypothesis based on the p-value and interpret this in the context of the original question.

Rubric

Your report should spell out all the steps you took in the process of answering these question: data import, data processing steps, justify any subsetting of the data and show any exploratory graphics that aid the discussion. Output without discussion will not achieve as many marks as output with discussion. Marks will be given out of a possible 10 marks with the following rubric.

Full Marks (10 marks)

  • An exceptionally well written, concise yet comprehensive report with very good discussion of results and carefully curated plots and/or tables.
  • No superfluous output.
  • Demonstrates creativity and flare by incorporating concepts/ideas/code that are beyond what was delivered in lectures and labs.
  • Representing complete or close to complete mastery of the material.

HD (9 marks)

  • A very well written report with good discussion of results and carefully curated plots and/or tables.
  • No superfluous output.
  • Representing complete or close to complete mastery of the material.

D (8 marks)

  • A comprehensive report, though it may include some unnecessary output. The discussion of results could be improved.
  • Representing excellence, but substantially less than complete mastery.

CR (7 marks)

  • A credit worthy attempt. These reports might not have provided sufficient evidence (graphical and formal hypothesis tests) to answer the underlying questions. The discussion of results could be improved.
  • Representing a creditable performance that goes beyond routine knowledge and understanding, but less than excellence.

P (6 marks)

  • A basic attempt at a report that insufficiently addresses all the questions.
  • Representing at least routine knowledge and understanding over a spectrum of topics and important ideas and concepts in the course.

F+ (4 marks)

  • A limited attempt at a report (e.g. incomplete)
  • When you don’t meet the learning outcomes to a satisfactory standard.

F (2 mark)

  • An unsatisfactory attempt or only an Rmd file submitted.

Z (0 marks)

  • Not a valid submission.

Feedback

When we ran a similar assessment task last year, this was the generic feedback we provided to all students:

Specific comments:

  1. Most people interpreted their p-values correctly.
  2. Structure of hypothesis testing was good.
  3. Most people clearly stated which hypothesis test was being performed each time.
  4. Be selective in which code output you show. For example, do you need to show the entire matrix of expected cell counts in the report?
  5. Be sure to write some text around whether the assumptions were satisfied (above and beyond just stating that what the assumptions are). For example, don’t rely on an output TRUE from the code and expect the reader to know what that means.
  6. Some people discretised continuous variables. When discretising continuous variables, you need to be clear about how the thresholds were determined. It’s also worth considering if your results robust to changes in those thresholds. Do you come to a different conclusion if you use slightly different cut-offs?
  7. What does Fisher’s test do in the case where the contingency table is larger than 2x2? If you don’t know (we didn’t cover it in lectures), best to avoid the test.

Where people lost marks:

  • Lost marks if they didn’t check the assumptions of the test, this was the most common mistake. Many students simply stated the assumptions without commenting on its validity.
  • Lost a mark if they didn’t answer each of the questions (use heading to make it clear to the marker). A few submissions were penalised if they came up with trivial questions like “What was the most common favourite season” - it needs to be a proper hypothesis test, not just an observation.
  • Lost a mark if they didn’t remove nonsensical values or otherwise didn’t clean the data before testing or didn’t talk about how the data was cleaned.
  • Many submissions performed multiple tests (e.g. a t-test and a Wilcoxon test) for the same question. This is OK if the assumptions are borderline (i.e. almost normally distributed data) but if the assumptions are clearly violated there is no need to do the t-test, just go with the valid test.
  • Related to the above: make sure you justify the choice of the test you use.
  • Need to see introductions and conclusions as well as list of references (even if it’s just the R packages used). Software package references are just as important as citing articles, websites or books.

General comments:

  • It’s always a good idea to visualise your data. If you’re doing a hypothesis test, there is usually some sort of visualisation that can accompany the hypothesis test so we can see what we’re testing.
  • Plots deserve to be treated with respect - nice labels, legend, etc. Not all plots need to be hypercoloured - there were some fine examples of rainbow bar charts, but it’s not really necessary to make each year group a different colour for example.
  • Most statisticians have a visceral hatred of pie charts. https://www.richardhollins.com/blog/why-pie-charts-suck/
  • If your factor (categorical) variables have a natural ordering, make sure this is maintained in the plot (e.g. Always, Often, Sometimes, Rarely, Never). If there’s no natural ordering, bar charts are best ordered by frequency.
  • You can include maths using \(\LaTeX\) if you know how.
  • Be sure to use message = FALSE and warning = FALSE in chunk options to prevent spurious messages and warnings showing up in the body of your report.
  • Your comments are just as important (if not more important) than your code. Interpreting your output for your audience is your primary job as a data scientist.
  • Anyone should be able to read your report and understand what is going on. (Not just statisticians.)
  • Presentation matters. Be sure to put some effort into defining your document structure with headings, subheadings, etc. It’s always nice to have an introduction or executive summary up front and a conclusion that outlines any limitations.
  • Some people put a lot of text as comments in their code - in an R markdown document, your comments can be outside the code chunks and they will appear as nicely formatted text.
  • RStudio has a spell check - use it!
  • The kable() function from the knitr package can be used to make tables look way nicer than monospaced font output. Also the gt package makes some nice tables.
  • The best submissions had clearly explored ways to do things that we haven’t necessarily covered in lectures and labs - this is the best way to learn: identify that you need to do something and google the problem. Learning to code independently is basically learning how to google search the right key words/phrases and doing lots of trial and error.
  • Careful with the use of the word significantly in colloquial text. The word “significant” has special meaning in a statistics context and should only be used with reference to a hypothesis test. Alternative words you can use to highlight an observed difference in more general settings include markedly, considerably, conspicuously, strikingly, …
  • Try not to hard code data e.g. don’t use something like c(45,12,9) to represent observed cell counts - one of the major advantages of having R markdown document is that it can auto update when the data set changed. The text can even be made to be adaptive, but at a minimum the models should be adaptive and then only the text needs to be updated each time the data set changes.
